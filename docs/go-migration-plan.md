# Go migration plan

Phased plan for completing the Python → Go rewrite on `feat/migrate-to-golang`.
Each phase is designed to be executed in **one short Claude Code session** with
minimal context, because the executor is on a usage-limited plan. Phases are
ordered by dependency; do them in order unless marked independent.

## How to execute a phase (instructions to the agent)

1. Read **only** the files listed under "Read" for that phase. Do not explore
   the rest of the repo — this plan plus CLAUDE.md is your context.
2. Port behavior faithfully from the Python source. The Python implementation
   is the spec; match its output byte-for-byte where it generates text
   (snippets, configs, headers), including marker lines and generated-file
   headers.
3. Port the listed Python tests to Go table-driven tests in the same package
   (`foo_test.go`). Use `t.TempDir()` for filesystem tests; never touch the
   real `$HOME`.
4. Verify: `go build ./...` then `go test ./...`. Both must pass.
5. Commit with the conventional message given for the phase. One commit per
   phase. Do not commit if tests fail.
6. Stop after the phase. Do not start the next phase in the same session.

### Standing rules (from CLAUDE.md, repeated here so the agent need not rediscover them)

- No third-party deps beyond `cobra` and `BurntSushi/toml`. `age` and `git`
  are invoked as external binaries via `os/exec` (matching Python).
- Errors: always return `*errs.DotsError` (or wrap) with a `Hint`. Copy hint
  text from the Python source.
- Idempotency: running any operation twice yields the same result.
- `--dry-run` produces zero side effects.
- Generated files get a "generated by dots" header; user files are backed up
  (`fileutil.Backup`) before overwrite.
- Reuse existing helpers in `internal/fileutil` (expand, sha256, backup,
  ensure-parent) instead of reimplementing them. Read `internal/fileutil/fileutil.go`
  first in every phase to see what already exists; add missing helpers there
  (e.g. marker-block insert/remove belongs in `internal/shell`, `run`-style
  exec wrapper can live in a new `internal/execx` or in `fileutil` — pick one
  in Phase 1 and stay consistent).

### Already done (do not redo)

- `internal/platform` — implemented.
- `internal/errs` — implemented.
- `internal/config` — `Load`, `FindRepoRoot`, full dots.toml parsing, tested.
- `internal/fileutil` — expand/sha256/backup helpers, tested.
- `internal/discovery` — implemented + tested.
- `internal/deploy` — symlink/copy/render + results, implemented + tested.
- `cmd/dots/commands.go` — all cobra command constructors exist; `apply`,
  `preview`, `status`, `diff`, `list` are wired to deploy/discovery. Subsystem
  commands (shell/git/ssh/tools/repos/env/presets/encrypt/decrypt/doctor/migrate)
  still call stub packages.

## Phase 0 — commit work in progress

## Phase 9 — parity check + Python removal

**Goal:** prove parity, then drop Python.

- Run the Python integration suite mentally against the Go binary: for each
  file in `tests/integration/`, confirm a Go equivalent exists; port any
  gaps.
- Manual smoke: `just build`, then `bin/dots --help`, `dots apply --dry-run`
  against a fixture repo, `dots doctor`.
- Remove `src/dots/`, `tests/` (Python), `pyproject.toml`; update `justfile`
  and `install.sh` for Go-only; update `release-please-config.json` if it
  references Python packaging.
- Commit: `feat!: complete Go migration, remove Python implementation`
  (check ROADMAP.md first — if the Python removal should wait for a release,
  split into `feat: ...` now and removal later).

## Post-migration fixes (review of 2026-06-11)

All 10 original phases are committed and `go build/vet/test` are green, but a
code review found bugs, parity gaps, and invariant violations. Fix them in the
phases below — same rules as above: read **only** the listed files, one commit
per phase, `go build ./... && go test ./...` green before committing, stop
after the phase.

### Phase 11 — correctness bugs (one session, six small fixes)

Read: `internal/shell/shell.go`, `internal/ssh/ssh.go`, `cmd/dots/main.go`,
`internal/tools/tools.go`, `cmd/dots/commands.go` (only `runApply`,
`printResults`, `applyPresets`), `internal/shell/shell_test.go`,
`internal/ssh/ssh_test.go`.

1. **Tool-snippet guard is wrong** — `internal/shell/shell.go:135-141`
   (`GenerateToolSnippet`): when `tool.Check` starts with `"which "`, the
   guard is set to `parts[0]` which is the literal string `which`, so
   `command -v which` always succeeds. Use `parts[1]` (the binary name).
   Add a test case: `Check: "which zoxide"` → snippet contains
   `command -v zoxide`.
2. **SSH config output is non-deterministic** — `internal/ssh/ssh.go:78`:
   `for k, v := range host.Options` iterates a map in random order, so
   `managed.gitconfig`'s sibling SSH fragment differs run-to-run (breaks
   invariant 3, idempotency). Collect keys, `sort.Strings`, then emit. Add a
   test generating the config twice for a host with ≥3 options and asserting
   byte equality.
3. **Non-DotsError errors are silent** — `cmd/dots/main.go:13-20`: the root
   command has `SilenceErrors: true` and main only prints when
   `errs.Unwrap(err)` matches a dots error type. A plain error (e.g. an
   `os.MkdirAll` failure bubbling up from `runInit`) exits 1 with **no output
   at all**. Add an `else` branch: `fmt.Fprintf(os.Stderr, "\n✗ %v\n", err)`.
   Also delete the dead `todo()` helper at the bottom of main.go.
4. **GitHub asset download ignores HTTP status** — `internal/tools/tools.go`
   `githubDownloadAsset`: a 404/403 response body (JSON error) is written to
   disk and later installed as the binary. After `httpClient.Do`, return a
   `errs.NewTool` error when `resp.StatusCode != http.StatusOK` (hint: include
   the status code and URL). Add a test using the existing `httpClient`/
   `githubAPIBase` test seams with an httptest server returning 404.
5. **Unknown archive types installed raw** — `internal/tools/tools.go`
   `installGitHub` switch: only `.tar.gz/.tgz/.zip` are extracted; a
   `.tar.xz`/`.tar.bz2`/`.tar.zst` asset falls through to `installBinaryFile`
   and the compressed archive itself is installed as the "binary". Before the
   default branch, detect these suffixes and return `errs.NewTool` with hint
   "Use the 'asset' field to select a .tar.gz or .zip asset." Test with a
   fake asset named `foo.tar.xz`.
6. **`dots apply` exits 0 on deploy errors** — `cmd/dots/commands.go`
   `runApply`/`printResults`: per-file errors are printed to stderr and
   counted but never affect the return value. Make `printResults` return the
   error count and have `runApply` return
   `errs.New(fmt.Sprintf("%d file(s) failed to deploy", n), "See errors above.")`
   when n > 0 (after running the remaining subsystems is NOT required —
   returning early after printResults is fine and simpler).

Commit: `fix: correct tool guard, ssh ordering, silent errors, download checks`

### Phase 12 — `dots shell clean` deletes the fzf preset + write hygiene

Read: `cmd/dots/commands.go` (`applyPresets`, `applyLoginShell`),
`internal/shell/shell.go` (`Clean`), `internal/fileutil/fileutil.go`,
`internal/shell/shell_test.go`, `cmd/dots/commands_test.go`.

1. **Clean/apply disagreement**: `applyPresets` writes `030-fzf.sh` into
   shell.d, but `shell.Clean`'s `expected` map only contains `010-env.sh`,
   `020-path.sh`, `050-*.sh`, `099-custom.sh` — so `dots shell clean` deletes
   the preset that `dots apply` just wrote. Fix: add `"030-fzf.sh"` to
   `expected` when `cfg.Presets.Fzf` is true (and keep the two in sync — add
   a comment in both places pointing at the other). Test: apply-style write
   then Clean, assert the file survives.
2. **User files overwritten without backup** (invariant 8) and rewritten when
   unchanged (invariant 3): `applyPresets` (tmux → `~/.tmux.conf`) and
   `applyLoginShell` (→ `~/.zprofile`, `~/.profile`) call `os.WriteFile`
   unconditionally. For each target: read existing content; if identical, skip
   (and print nothing or "unchanged"); if the existing file does **not** start
   with a dots generated header (`shell.GeneratedHeader` or the
   `# Generated by dots preset:` prefix), call `fileutil.Backup` before
   writing. The fzf snippet lives inside shell.d (dots-owned), no backup
   needed there.
3. `internal/tools/tools.go` defines a private `expandHome` duplicating
   `fileutil.Expand` — delete it and use `fileutil.Expand` (CLAUDE.md:
   never reimplement helpers).

Commit: `fix: stop shell clean removing presets; back up user files on overwrite`

### Phase 13 — files.d platform discovery + Platforms() threading

The biggest parity gap. ADR 004 specifies `files.d/{platform}/` scoping;
`dots init` creates `files.d/`, `dots migrate --platform` writes into it, and
docs/platforms.md describes it — but `discovery.Walk` never reads it (its own
doc comment admits "not yet implemented"). Separately, commit ff58445 added
`platform.Platforms()` (multi-tag, e.g. `["linux","wsl"]`) but only
`dots env check` uses it; deploy filtering, discovery, ssh host filtering, and
env-snippet generation still match against the single `platform.Detect()` tag,
so `only = ["wsl"]` entries never apply anywhere that matters — and
`dots env check` (multi-tag) can report ✓ for a var that
`shell.GenerateEnvSnippet` (single-tag) omits.

Read: `adr/004-files-d-platform-scoping.md`, `docs/platforms.md`,
`internal/discovery/discovery.go` + test, `internal/deploy/deploy.go`
(`Apply`/`Status` platform filter, `containsPlatform`, `currentPlatform`),
`internal/shell/shell.go` (`generateEnvSnippet` only),
`internal/ssh/ssh.go` (`GenerateConfig` only), `internal/platform/platform.go`,
call sites in `cmd/dots/commands.go` that pass `platform.Detect()`.

1. Change the platform parameter to `[]string` (the result of
   `platform.Platforms()`) in: `discovery.Walk`, the deploy `Options` (add
   `Platforms []string`; delete the unused `currentPlatform(repoRoot)`
   helper whose arg is already ignored), `ssh.GenerateConfig`/`WriteManaged`/
   `ShowManaged`, and `shell.generateEnvSnippet`. Matching rule everywhere:
   entry is active if `only` is empty or intersects the tag list.
2. In `discovery.Walk`, after walking `files/`, walk
   `files.d/<tag>/` for each tag in the platform list (the `walkDir` `only`
   parameter already exists for this — pass `[]string{tag}`). files.d entries
   override same-`Dst` entries from `files/` (later wins), and explicit
   `[[file]]` entries still override everything (existing `mergeEntries`).
   Follow ADR 004 for the precedence order if it differs — the ADR is the
   spec.
3. Port/extend tests: discovery test with `files.d/linux/` and `files.d/wsl/`
   fixtures; deploy filter test with `Only: ["wsl"]` and
   `Platforms: ["linux","wsl"]`; assert `env check` and
   `GenerateEnvSnippet` agree (same tag list in, same vars out).

Commit: `feat(discovery): implement files.d platform scoping with multi-tag matching`

### Phase 14 — deploy .age secrets (and decide on templates)

`deploy.Apply` (internal/deploy/deploy.go:37-39) silently skips any entry
with `Template` or `Secret` set, yet discovery flags `.age`/`.j2` files,
`secrets.DecryptToMemory` exists precisely "to render decrypted files
in-memory" and is never called, and CLAUDE.md claims deploy is
"symlink/copy/render — implemented".

Read: `internal/deploy/deploy.go`, `internal/secrets/secrets.go`,
`internal/discovery/discovery.go` (flagging only), `adr/008-age-for-secrets.md`,
`internal/deploy/deploy_test.go`, `internal/secrets/secrets_test.go`.

1. Secrets: in `deploy.Apply`, when `entry.Secret`, call
   `secrets.DecryptToMemory(src, cfg)` and write the plaintext to dst with
   mode `0600` (always copy mode — never symlink a decrypted secret into the
   repo). Unchanged-detection: compare plaintext hash against dst. On
   missing age/identity return the error in `Result.Err` (the hints are
   already in the secrets package). Dry-run: report "decrypt" action, no
   side effects, and do not require age to be installed. This needs cfg
   plumbed into `deploy.Options` (add `Config config.Config` or just
   `Secrets config.SecretsConfig` — prefer the smaller one).
   Gate real-age tests behind `exec.LookPath("age")` skip like
   secrets_test.go does.
2. Templates: `.j2` rendering was never specced in Go (no ADR). Do **not**
   implement Jinja2. Keep skipping `Template` entries but make it visible:
   report action "skipped (template — not supported)" once, and update
   CLAUDE.md + docs/architecture.md to say render means "secret decryption";
   templates are unsupported in the Go version. If ROADMAP.md mentions
   templates, leave it — this is a docs correction, not a feature decision.

Commit: `feat(deploy): decrypt .age secrets on apply`

### Phase 15 — idempotency + small cleanups

Read: `cmd/dots/commands.go` (`runAdd`, `runDoctor`, `newCompletionCmd`),
`cmd/dots/commands_test.go`.

1. **`dots add` is not idempotent** (invariant 3): running it twice appends a
   duplicate `[[file]]` block to dots.toml. Before appending, check whether
   `globals.cfg.Files` already contains an entry with the same src or dst
   (or the repo dest file already exists) and print "already managed" +
   return nil instead.
2. **doctor dead code**: `filepath.Glob(".../**/*.age")` does not recurse in
   Go — delete the Glob call and keep only the WalkDir fallback, and make
   WalkDir skip `.git` (`if d.IsDir() && d.Name() == ".git" { return
   filepath.SkipDir }`).
3. **doctor exit code**: warnings currently produce exit 1, same as errors.
   Decide: errors → 1, warnings only → 0 (doctor is advisory; matches
   common CLI convention). Update `runDoctor` and any test.
4. `newCompletionCmd`: drop the no-op `DisableFlagParsing: false` line and
   the unreachable-in-practice `default:` branch can stay (harmless) — just
   use `errs.New` instead of constructing `&errs.DotsError{}` directly, for
   consistency.

Commit: `fix(cli): make add idempotent, clean up doctor checks and exit codes`

### Known limitations (documented, not scheduled)

- tar extraction validates symlink entries but never creates them (only
  dirs and regular files are extracted). Fine for single-binary releases;
  revisit only if a real tool needs in-archive symlinks.
- `git.GenerateConfig` output depends on `exec.LookPath("delta")`, so
  `dots git show` differs between machines. Intentional (matches "only enable
  delta when present") — do not "fix".

## Session budget notes (for the human)

- Phases 1–4, 6, 8 are small (one module each, ~100–300 lines of Go +
  tests) — comfortably one Pro session each, likely on Sonnet.
- Phase 5 (tools) and Phase 7 (apply orchestration) are the big ones —
  consider Opus, fresh session, and use the built-in split point in Phase 5.
- Always `/clear` between phases; the plan file is the only carry-over
  context the agent needs.
- If a phase ends with failing tests and the session dies, the next session's
  first instruction should be: "Finish phase N of docs/go-migration-plan.md;
  tests currently failing, fix then commit."
