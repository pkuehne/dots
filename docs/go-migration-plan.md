# Go migration plan

Phased plan for completing the Python → Go rewrite on `feat/migrate-to-golang`.
Each phase is designed to be executed in **one short Claude Code session** with
minimal context, because the executor is on a usage-limited plan. Phases are
ordered by dependency; do them in order unless marked independent.

## How to execute a phase (instructions to the agent)

1. Read **only** the files listed under "Read" for that phase. Do not explore
   the rest of the repo — this plan plus CLAUDE.md is your context.
2. Port behavior faithfully from the Python source. The Python implementation
   is the spec; match its output byte-for-byte where it generates text
   (snippets, configs, headers), including marker lines and generated-file
   headers.
3. Port the listed Python tests to Go table-driven tests in the same package
   (`foo_test.go`). Use `t.TempDir()` for filesystem tests; never touch the
   real `$HOME`.
4. Verify: `go build ./...` then `go test ./...`. Both must pass.
5. Commit with the conventional message given for the phase. One commit per
   phase. Do not commit if tests fail.
6. Stop after the phase. Do not start the next phase in the same session.

### Standing rules (from CLAUDE.md, repeated here so the agent need not rediscover them)

- No third-party deps beyond `cobra` and `BurntSushi/toml`. `age` and `git`
  are invoked as external binaries via `os/exec` (matching Python).
- Errors: always return `*errs.DotsError` (or wrap) with a `Hint`. Copy hint
  text from the Python source.
- Idempotency: running any operation twice yields the same result.
- `--dry-run` produces zero side effects.
- Generated files get a "generated by dots" header; user files are backed up
  (`fileutil.Backup`) before overwrite.
- Reuse existing helpers in `internal/fileutil` (expand, sha256, backup,
  ensure-parent) instead of reimplementing them. Read `internal/fileutil/fileutil.go`
  first in every phase to see what already exists; add missing helpers there
  (e.g. marker-block insert/remove belongs in `internal/shell`, `run`-style
  exec wrapper can live in a new `internal/execx` or in `fileutil` — pick one
  in Phase 1 and stay consistent).

### Already done (do not redo)

- `internal/platform` — implemented.
- `internal/errs` — implemented.
- `internal/config` — `Load`, `FindRepoRoot`, full dots.toml parsing, tested.
- `internal/fileutil` — expand/sha256/backup helpers, tested.
- `internal/discovery` — implemented + tested.
- `internal/deploy` — symlink/copy/render + results, implemented + tested.
- `cmd/dots/commands.go` — all cobra command constructors exist; `apply`,
  `preview`, `status`, `diff`, `list` are wired to deploy/discovery. Subsystem
  commands (shell/git/ssh/tools/repos/env/presets/encrypt/decrypt/doctor/migrate)
  still call stub packages.

## Phase 0 — commit work in progress

**Goal:** get the working tree clean so each later phase is one atomic commit.

- Read: `git status` / `git diff` only.
- Action: review and commit the pending deploy/discovery/fileutil work
  (currently modified/untracked) as e.g.
  `feat(deploy): implement file deployment and discovery with tests`.
  Split into two commits (discovery, deploy) if the diff separates cleanly.
- Verify: `go test ./...` green, `git status` clean.

## Phase 1 — internal/shell

**Goal:** snippet generation + marker-delimited insertion. Foundation for
git/ssh/presets phases (they reuse the marker-block pattern and generated
headers).

- Read: `src/dots/shell.py`, `src/dots/utils.py`, `internal/shell/shell.go`,
  `internal/config/config.go` (Shell/Tool types only), `internal/fileutil/fileutil.go`,
  `tests/unit/test_snippets.py`, `adr/007-marker-delimited-insertion.md`.
- Implement in `internal/shell`: `GenerateEnvSnippet`, `GeneratePathSnippet`,
  `GenerateToolSnippet`, custom-snippet discovery (port
  `generate_custom_snippet`), `WriteSnippets`, `InsertSourceLine`
  (port `idempotent_insert`), `Clean` (port `remove_marker_block`),
  `Assembled`.
- Port tests: `tests/unit/test_snippets.py` → `internal/shell/shell_test.go`.
- Commit: `feat(shell): implement snippet generation and marker insertion`

## Phase 2 — internal/git + internal/ssh (one session, they're small twins)

**Goal:** managed gitconfig and ssh config generation.

- Read: `src/dots/git.py`, `src/dots/ssh.py`, `internal/git/git.go`,
  `internal/ssh/ssh.go`, relevant Config structs,
  `tests/unit/test_git_config.py`, `tests/unit/test_ssh_config.py`.
- Implement: `git.GenerateConfig/WriteManaged/ShowManaged/Uninit`,
  `ssh.GenerateConfig/WriteManaged/ShowManaged/Uninit` (incl.
  `snake_to_ssh_keyword` mapping, Include-line insertion in
  `~/.ssh/config` per `ssh_init`).
- Port both unit test files.
- Commit: `feat(git,ssh): implement managed gitconfig and ssh config generation`

## Phase 3 — internal/secrets

**Goal:** age encrypt/decrypt via `os/exec`.

- Read: `src/dots/secrets.py`, `internal/secrets/secrets.go`,
  `tests/integration/test_secrets.py`.
- Implement `Encrypt`, `Decrypt`, `DecryptToMemory` shelling out to `age`;
  copy the exact error hints (missing binary, missing identity, decrypt
  failure). Wire `newEncryptCmd`/`newDecryptCmd` in `cmd/dots/commands.go`
  (port `cmd_encrypt`/`cmd_decrypt` from `src/dots/commands.py:690-713`).
- Tests: unit-test the command construction and error paths with `age`
  absent; gate real round-trip tests behind `testing.Short()` /
  `exec.LookPath("age")` skip.
- Commit: `feat(secrets): implement age encrypt/decrypt`

## Phase 4 — internal/presets

**Goal:** preset generation (fzf, zprofile, profile) + eject.

- Read: `src/dots/presets.py`, `internal/presets/presets.go`,
  `adr/009-preset-eject-pattern.md`, `adr/010-profile-auto-activation.md`,
  `tests/unit/test_profiles.py` (profile parts).
- Implement `Generate`, `Eject`; output must match Python byte-for-byte.
- Commit: `feat(presets): implement preset generation and eject`

## Phase 5 — internal/tools

**Goal:** the biggest module — tool install via GitHub releases, plus other
install methods. Likely the longest phase; if a session runs low, stop after
the GitHub-release path + tests and finish remaining methods in a follow-up
session (split commits: `feat(tools): github release install`,
`feat(tools): remaining install methods`).

- Read: `src/dots/tools.py`, `internal/tools/tools.go`, Tool/ToolInstall
  config structs, `adr/006-github-releases-first.md`,
  `adr/013-arch-map-per-tool-override.md`, `tests/unit/test_tools.py` (the
  unit file, not the 450-line integration file — port integration tests only
  if budget allows).
- Implement: `Check` (installed detection), `Filter`, `Install` dispatch,
  GitHub latest-release lookup + asset selection (stdlib `net/http`,
  `encoding/json`), safe tar/zip extraction (port `_safe_tar_extractall` /
  `_safe_zip_extractall` path-escape checks — security-relevant, keep them),
  binary install into bin dir.
- Commit(s): see above.

## Phase 6 — internal/repos

**Goal:** repo clone/update/status via `git` subprocess.

- Read: `src/dots/repos.py`, `internal/repos/repos.go`,
  `tests/integration/test_repos.py`.
- Implement `Clone`, `Update`, `Status`, `Filter`. Tests can use local
  file:// remotes created in `t.TempDir()` (mirror the Python integration
  tests).
- Commit: `feat(repos): implement repo cloning and updating`

## Phase 7 — CLI wiring: subsystem dispatch + apply orchestration

**Goal:** make every cobra command call the now-real packages, and port the
`cmd_apply` orchestration order.

- Read: `src/dots/commands.py` (`cmd_apply`, `_apply_shell`, `_apply_git`,
  `_apply_repos`, `_apply_presets`, `_apply_login_shell`, `_apply_tools`,
  `dispatch_*`), `cmd/dots/commands.go`, `tests/integration/test_apply.py`.
- Wire: shell/git/ssh/env/tools/repos/presets subcommands; extend `apply` to
  run subsystems in the same order as Python. Honor `--dry-run` everywhere.
- Port the apply integration tests (`internal/` package test or a
  `cmd/dots` test driving the cobra root with a temp HOME).
- Commit: `feat(cli): wire subsystem commands and full apply orchestration`

## Phase 8 — doctor, migrate, init, edit, add

**Goal:** remaining commands.

- Read: `src/dots/commands.py` (`cmd_doctor`, `cmd_migrate`, `cmd_init`,
  `cmd_edit`, `cmd_add`), `cmd/dots/commands.go`.
- Implement and wire; doctor checks should cover every subsystem implemented
  above.
- Commit: `feat(cli): implement doctor, migrate, init, edit and add`

## Phase 9 — parity check + Python removal

**Goal:** prove parity, then drop Python.

- Run the Python integration suite mentally against the Go binary: for each
  file in `tests/integration/`, confirm a Go equivalent exists; port any
  gaps.
- Manual smoke: `just build`, then `bin/dots --help`, `dots apply --dry-run`
  against a fixture repo, `dots doctor`.
- Remove `src/dots/`, `tests/` (Python), `pyproject.toml`; update `justfile`
  and `install.sh` for Go-only; update `release-please-config.json` if it
  references Python packaging.
- Commit: `feat!: complete Go migration, remove Python implementation`
  (check ROADMAP.md first — if the Python removal should wait for a release,
  split into `feat: ...` now and removal later).

## Phase 10 — docs + ADRs

**Goal:** documentation reflects the Go implementation.

- Write `adr/014-go-rewrite.md`: motivation (single static binary, no Python
  runtime dependency), consequences, and note that it **supersedes**
  `adr/001-single-python-file.md`, `adr/011-package-split.md`, and
  `adr/012-minimum-python-version.md`. Add a "Superseded by ADR-014" status
  line to those three ADRs (do not delete them).
- Update `docs/architecture.md` (package layout = `cmd/` + `internal/`),
  `docs/contributing.md` (Go toolchain, just commands),
  `docs/migration.md`, `docs/troubleshooting.md` (error/hint examples),
  `docs/configuration.md` (only if behavior diverged — it shouldn't),
  `README.md` (install instructions), and CLAUDE.md (drop the Python
  reference sections, mark stubs as implemented, rewrite the "Adding a new
  managed subsystem" checklist in Go terms).
- Commit: `docs: update docs and ADRs for the Go rewrite`

## Session budget notes (for the human)

- Phases 1–4, 6, 8 are small (one module each, ~100–300 lines of Go +
  tests) — comfortably one Pro session each, likely on Sonnet.
- Phase 5 (tools) and Phase 7 (apply orchestration) are the big ones —
  consider Opus, fresh session, and use the built-in split point in Phase 5.
- Always `/clear` between phases; the plan file is the only carry-over
  context the agent needs.
- If a phase ends with failing tests and the session dies, the next session's
  first instruction should be: "Finish phase N of docs/go-migration-plan.md;
  tests currently failing, fix then commit."
